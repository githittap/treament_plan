## 기술 스택

- **HTML5 Canvas** — 이미지·마커·텍스트·영역 렌더링

- **배포** — GitHub Pages (push → 자동 배포)

## 핵심 데이터 구조

```js
// 주석(annotation) 오브젝트 공통 필드
{
  id,        // 고유 번호 (idSeq++)
  type,      // 'tooth' | 'area' | 'text' | 'free'
  treatId,   // treatTypes[].id 참조
  x, y,      // 캔버스 좌표
  toothNum,  // FDI 번호 (예: "26")
  note, kinds, periods, surface,
  rotation, size,
  noLabel    // true = 치식 라벨 안 그림 (스티커류)
}

// treatTypes 배열 — 치료 종류 정의
{ id, label, shape, color, abbr, stroke, fill, skipPopup, interaction }

// GRID_COLUMNS — 치료종류 UI 배치
// '---' = 구분선, ['id1','id2'] = 한 열
const GRID_COLUMNS = [
  ['implant','pontic'], ['resin','inlay','fu'],
  ['core_crown'], ['ca','other'],
  '---',
  ['ongoing','done','hold']
];
```

## Karpathy 가이드라인 (이 프로젝트 적용판)

> **트레이드오프:** 이 가이드라인은 속도보다 신중함을 우선합니다. 사소한 작업은 판단하여 적용하세요.

### 1. 코딩 전 먼저 생각하기

**가정하지 말 것. 혼란을 숨기지 말 것. 트레이드오프를 드러낼 것.**

- 가정은 명시적으로 밝힐 것. 불확실하면 물어볼 것
- 해석이 여러 가지라면 선택지를 먼저 제시하고 확인 후 진행
- 단일 파일 구조이므로, 기능 추가 전 **기존 함수·변수와 충돌 여부** 반드시 확인
- 더 단순한 접근이 있다면 말할 것. 필요하면 **반박할 것**
- 치식 번호 체계(FDI), 치료종류 ID, 스티커 vs 팝업 흐름이 헷갈리면 **작업 멈추고** 무엇이 헷갈리는지 명시한 뒤 물어볼 것

### 2. 단순함 우선

**요청된 것만 구현. 추측성 코드 금지.**

- 요청하지 않은 기능, 추상화, 설정 옵션 추가 금지
- 불가능한 시나리오에 대한 에러 처리 추가 금지
- 200줄로 쓸 수 있는 걸 50줄로 쓸 수 있다면 다시 쓸 것
- 단일 파일이므로 파일 분리·번들링 제안 금지 (요청 시에만)
- 자기점검: **"시니어 엔지니어라면 이게 과하다고 할까?"** → 그렇다면 단순화

### 3. 외과적 변경

**꼭 바꿔야 할 것만 건드릴 것. 내가 만든 쓰레기만 치울 것.**

- 기존 함수 스타일, 들여쓰기, 변수명 규칙 유지 — **내 방식과 달라도** 기존 스타일을 따를 것
- 요청과 무관한 코드 리팩터링·포맷팅 금지
- 관련 없는 데드코드 발견 시 → 삭제하지 말고 언급만

**내 변경이 만든 고아(orphan) 처리:**
- 내 수정으로 인해 쓰이지 않게 된 import·변수·함수 → 직접 삭제
- 원래부터 있던 데드코드 → 삭제 금지, 언급만

변경 테스트: **수정된 모든 줄이 사용자 요청에서 직접 추적 가능해야 함**

### 4. 목표 중심 실행

**성공 기준을 정의하고, 검증될 때까지 반복.**

작업을 검증 가능한 목표로 변환:
- "ESC 동작 변경" → "팝업 열린 상태에서 ESC: 팝업 닫힘 / 마커 선택 상태에서 ESC: 마커 삭제됨"
- "RCT 스티커 이동" → "치료종류 그리드에서 RCT 사라짐 / 표시도구에 RCT 나타남 / V키로 배치 가능"

여러 단계 작업 전 다음 형식으로 계획 제시:

```
1. [변경 내용] → 검증: [확인 방법]
2. [변경 내용] → 검증: [확인 방법]
```

명확한 성공 기준 = 반복 독립 실행 가능. 모호한 기준("되게 해줘") = 계속 재확인 필요.

### 난독화 빌드 (보유만, 현재 미배포)

카피 방지용 난독화 도구를 만들어 두었으나 **현재 배포에는 사용하지 않음**.

```
node build_obfuscate.js   # 치료계획.html → 치료계획.min.html (난독화본) 생성
```

- `build_obfuscate.js` — 메인 `<script>`만 난독화 (변수명 mangling, 문자열 base64, 제어흐름 평탄화, selfDefending 등)
- `치료계획.min.html` — 자동 생성물. **직접 수정 금지** (수정은 항상 원본 `치료계획.html`에서)
- 의존성: `npm install javascript-obfuscator` (`node_modules`는 .gitignore 처리됨)
- 나중에 난독화본을 배포하려면 → 원본을 `치료계획.src.html`로 보관하고 빌드 출력을 `치료계획.html`로 바꾸는 방식 권장 (URL 유지)

저작권 표기: HTML 상단 주석 + 화면 하단 푸터에 `© 2026 Jung · 무단 복제·배포 금지`.

## 주의사항

- `redoStack=[]` — 새 annotation 추가 시 반드시 초기화할 것
- `_dblClickFired` — 더블클릭 시 pointerdown 중복 방지 플래그 (300ms)
- `deletedImageState` — 이미지 삭제 시 undo용 스냅샷
- `baseUnit()` = `Math.min(naturalW, naturalH) / 10` — 캔버스 크기 기준 단위
- `noLabel:true` — 스티커(발치·RCT·GBR 등)는 치식 라벨 안 그림
- 한글 파일명(`치료계획.html`) — git에서 UTF-8 인코딩으로 처리됨
